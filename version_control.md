#  售后AI机器人 — 版本管理改造任务说明

> 本文档是给 Claude Code 的任务规格说明，描述如何在现有 Qdrant RAG 系统中新增**产品版本管理**能力。请仔细阅读全部内容后再开始编码。

---

## 一、业务背景

我们的售后AI机器人使用 RAG 架构，知识库中包含三类文档：
- **产品说明书**（product_spec）
- **产品用户手册**（user_manual）— 写的是通用/基础版本
- **测试报告**（test_report）— 写的是各版本新增的差异内容

### 版本关系规则

| 规则 | 说明                           |
|------|------------------------------|
| 基础版本 | 如 `1101`，用户手册描述的是该版本的全部功能    |
| 特殊版本 | 如 `1102`、`1103`，基于1101版本的独立分支 |
| 分支关系 | 特殊版本之间**互不包含**，都独立继承自基础版本    |
| 冲突规则 | 当特殊版本内容与基础版本冲突时，**以特殊版本为准**  |
| 产品线 | 多产品线，但版本号全局唯一，无需 product_id 来消歧 |

### 版本号规则介绍

-不同产品的版本号规则不同，以LT-BPro产品为例，基础版本是1101开头的，后面的数字累加，如1101的第十个版本，就是110110V，以此类推
-同理，看到版本号为110171V，就知道是BPro产品的基础版本的第71个版本。
### 用户交互规则

- 用户知道自己使用的版本号，但**不一定主动说**
- 系统需在对话开始或首次提问时**引导用户提供版本号**
- 获得版本号后再执行检索

## 四、各层详细需求

### 4.1 版本关系管理层（新增模块）

#### 目的

提供一个持久化的版本关系映射，后期交由售后人员维护，**不能是配置文件**。

#### 需求

1. **新建数据库表** `version_registry`（使用项目现有的关系型数据库，如 PostgreSQL / MySQL / SQLite 均可，视项目现状而定）：

   | 字段 | 类型 | 说明 |
   |------|------|------|
   | `id` | int, PK, 自增 | 主键 |
   | `version_code` | varchar, unique | 版本号，如 `"1101"`、`"1102"` |
   | `version_name` | varchar | 版本显示名称（可选，方便售后辨识） |
   | `is_base` | boolean | 是否为基础版本 |
   | `base_version_code` | varchar, nullable | 对应的基础版本号。基础版本自身此字段为 null |
   | `doc_type_label` | varchar | 该版本关联的文档业务类型（`user_manual` / `test_report` / `product_spec`）—— 此字段可选，视是否需要细化 |
   | `created_at` | datetime | 创建时间 |
   | `updated_at` | datetime | 更新时间 |

2. **提供 CRUD API 接口**（集成到现有管理后台均可），使售后人员可以：
   - 查看所有版本及其关系
   - 新增版本并指定其基础版本
   - 修改版本关系
   - 删除版本记录

3. **提供一个查询函数** `get_base_version(version_code: str) -> str | None`：
   - 输入特殊版本号 → 返回其基础版本号
   - 输入基础版本号 → 返回 None（表示自身即基础版）
   - 输入不存在的版本号 → 抛出异常或返回 None

4. **提供一个查询函数** `get_all_versions() -> list`，返回所有版本列表，用于前端下拉框或对话中引导用户选择。

5. **提供一个查询函数** `is_base_version(version_code: str) -> bool`。

### 4.2 入库层改造

#### 目的

在每个 chunk 存入 Qdrant 时，payload 中附加版本相关的元数据。

#### 需求

1. **在 `indexer.py` 的 metadata 字典中新增以下字段**：

   | 新增字段 | 类型 | 说明 |
   |----------|------|------|
   | `version` | string | 该 chunk 所属的版本号，如 `"1101"` |
   | `doc_type` | string | 业务文档类型：`user_manual` / `test_report` / `product_spec` |

   > 注意：`is_base` 和 `base_version` 不需要冗余存到每个 chunk 中，检索时从 `version_registry` 表实时查询即可，避免数据不一致。

2. **版本号和文档类型的来源**：
   - 方案 A（推荐）：在文档上传/入库流程中，要求上传者手动指定版本号和文档类型（通过上传界面的表单字段）
   - 方案 B：从文件名中通过正则提取（如文件名包含版本号）
   - 方案 C：从文档内容中自动解析（不推荐，准确率不可控）
   - **请先查看现有上传流程的代码，选择最合适的方案。优先选方案 A。**

3. **为新字段创建 Qdrant payload 索引**：
   ```python
   # 需要新增的 payload 索引
   qdrant_client.create_payload_index(
       collection_name="index_document_elements",
       field_name="metadata.version",
       field_schema="keyword"
   )
   qdrant_client.create_payload_index(
       collection_name="index_document_elements",
       field_name="metadata.doc_type",
       field_schema="keyword"
   )
   ```

4. **数据迁移**：对于已入库的旧数据，提供一个迁移脚本：
   - 遍历 Qdrant 中所有现有 point
   - 根据 `source`（文件名）识别版本号和文档类型（可能需要我提供映射规则或手动标注）
   - 批量更新 payload
   - **迁移脚本单独写一个文件，不要混入主流程代码**

### 4.3 检索层改造

#### 目的

根据用户的版本号，执行**两阶段检索**，并合理合并结果。

#### 检索流程

```
用户提问 + 版本号 version_code
        │
        ▼
┌─ 查询 version_registry ──────────────────────────┐
│  get_base_version(version_code)                   │
│  → 如果返回 None：说明是基础版本，只做一阶段检索     │
│  → 如果返回 base_code：执行两阶段检索              │
└───────────────────────────────────────────────────┘
        │
        ▼ （两阶段检索的情况）
┌─ 阶段1：特殊版本检索 ─────────────────────────────┐
│  向量相似度搜索                                     │
│  + 过滤条件：metadata.version == version_code      │
│  → 取 top_k 个结果（如 top_k=5）                   │
└───────────────────────────────────────────────────┘
        │
        ▼
┌─ 阶段2：基础版本补充检索 ─────────────────────────┐
│  向量相似度搜索                                     │
│  + 过滤条件：metadata.version == base_code         │
│  → 取 top_k 个结果（如 top_k=5）                   │
└───────────────────────────────────────────────────┘
        │
        ▼
┌─ 结果合并 ────────────────────────────────────────┐
│  1. 特殊版本结果在前，标记 priority = "high"        │
│  2. 基础版本结果在后，标记 priority = "low"         │
│  3. 每条结果附带元数据标签，传给 LLM                 │
└───────────────────────────────────────────────────┘
```

#### 具体要求

1. **修改现有的检索函数**，新增 `version_code` 参数
2. **使用 Qdrant 的 `Filter` 实现过滤**，示例：
   ```python
   from qdrant_client.models import Filter, FieldCondition, MatchValue

   version_filter = Filter(
       must=[
           FieldCondition(
               key="metadata.version",
               match=MatchValue(value=version_code)
           )
       ]
   )
   ```
3. **合并后的上下文格式**：每条 chunk 送给 LLM 时，前面加上来源标记：
   ```
   [版本:{version} | 文档类型:{doc_type} | 优先级:{priority} | 来源:{source}]
   {original_content}
   ```
4. **如果用户查询的是基础版本**，则只做一次检索（过滤 `version == base_code`），无需两阶段。

### 4.4 对话层改造

#### 目的

确保用户版本已知后再执行检索，并让 LLM 正确处理冲突。

#### 需求 1：版本确认前置

在对话管理中维护一个会话级状态 `current_version`：
- 每次用户提问时，检查 `current_version` 是否已设置
- 如未设置 → **不执行检索**，先回复引导用户提供版本号（可提供版本列表供选择）
- 已设置 → 正常执行检索流程
- 用户可随时切换版本（如"我换成1103了"），更新 `current_version`

#### 需求 2：System Prompt 改造

在发送给 LLM 的 system prompt 中，新增以下规则（合并到现有 prompt 中）：

```
## 版本知识检索规则

你收到的参考资料中，每条内容前都有来源标记，格式为：
[版本:XXX | 文档类型:XXX | 优先级:XXX | 来源:XXX]

请遵守以下规则：
1. 优先级为 "high" 的内容来自用户当前使用的特殊版本，优先级为 "low" 的内容来自基础版本。
2. 当高优先级和低优先级内容描述同一功能但存在差异时，**以高优先级内容为准**。
3. 当只有低优先级内容涉及某功能时，正常使用该内容回答。
4. 回答时，如果使用了特殊版本的内容覆盖了基础版本，请简要说明。
5. 不要向用户暴露"优先级"、"high/low"等内部标记术语。
```

## 六、注意事项

1. **不要改变现有 chunk 的分块逻辑和向量化逻辑**，只在 metadata 层面做增强
2. **version_registry 必须持久化在关系型数据库中**（不是配置文件、不是环境变量），因为后续要交给售后人员通过界面维护
3. **每一步改造完成后，确保现有功能不受影响**（向后兼容）—— 旧数据中没有 `version` 字段的 chunk，检索时应跳过过滤或走降级逻辑
4. **先阅读现有代码结构**，特别是 入库流程、检索函数、对话管理模块，理解当前实现后再动手
5. 有任何不确定的地方，先问我，不要自行假设
