# 未知问题处理机制分析

## 问题背景

当用户提出数据库中没有记录的问题时，系统需要优雅地处理这种情况，既不能给出错误或无用的信息，也不能假装知道答案。本文从**向量模型**和**大语言模型**两个维度分析系统的处理机制。

## 系统架构

```
用户查询
    ↓
[1. 向量检索层] - 向量模型
    ↓
检索结果 + 置信度
    ↓
[2. 答案生成层] - 大语言模型
    ↓
最终回答
```

---

## 一、向量模型层面的处理

### 1.1 工作原理

向量模型(text-embedding-v4)将文本转换为高维向量，通过**余弦相似度**计算语义相似性。

```python
# 核心计算
query_embedding = get_embeddings(query)  # [768维向量]
similarities = dot(knowledge_embeddings, query_embedding)  # 计算相似度
```

### 1.2 相似度阈值机制

系统定义了三个置信度级别:

| 置信度 | 相似度范围 | 含义 |
|-------|-----------|------|
| `high` | ≥ 0.75 | 高度匹配，知识库有明确答案 |
| `medium` | 0.45 - 0.75 | 中等匹配，有相关信息 |
| `low` | < 0.45 | 低匹配，几乎无相关信息 |

### 1.3 未知问题的处理行为

#### 测试案例1: "充电桩的WiFi密码忘记了"
```
向量检索结果:
  置信度: medium
  最佳匹配: [E10] 充电桩无法启动充电
  相似度: 0.5604
```

**分析**:
- ❌ 问题: WiFi密码问题 (数据库中不存在)
- ✓ 匹配: 充电桩相关问题
- 📊 相似度: 0.56 (中等)
- 🎯 行为: 向量模型抓住了"充电桩"关键词，找到相关但不精确的答案

#### 测试案例2: "加油机可以用来充电吗"
```
向量检索结果:
  置信度: medium
  最佳匹配: [E01] 加油枪无法出油
  相似度: 0.6148
```

**分析**:
- ❌ 问题: 概念性问题 (非故障)
- ✓ 匹配: 加油机相关
- 📊 相似度: 0.61 (中等)
- 🎯 行为: 向量匹配到"加油机"关键词，但问题类型不同

#### 测试案例3: "设备屏幕突然变成紫色"
```
向量检索结果:
  置信度: medium
  最佳匹配: [E25] 显示板检测错误
  相似度: 0.4673
```

**分析**:
- ❌ 问题: 紫色屏幕 (特定症状，数据库无)
- ✓ 匹配: 显示板问题
- 📊 相似度: 0.47 (接近阈值)
- 🎯 行为: 语义理解"屏幕"→"显示板"，但症状不匹配

#### 测试案例4: "E99错误代码"
```
向量检索结果:
  置信度: medium
  最佳匹配: [E26] 与后台通讯错误
  相似度: 0.6049
```

**分析**:
- ❌ 问题: E99 (不存在的错误代码)
- ✓ 匹配: 其他错误代码
- 📊 相似度: 0.60 (中等)
- 🎯 行为: 找到结构相似的错误代码条目

### 1.4 向量模型的优势与局限

#### ✅ 优势

1. **语义理解**: 能理解"屏幕"≈"显示板"
2. **关键词捕获**: 抓住"充电桩"、"加油机"等核心概念
3. **模糊匹配**: 即使问题不精确，也能找到相关信息
4. **永不空手**: 总能返回某些结果(只要相似度>0.45)

#### ❌ 局限

1. **语义漂移**: WiFi密码 → 充电桩启动问题 (关联但不相关)
2. **无法判断问题类型**: 概念性问题 vs 故障问题
3. **相似度不可靠**: 0.56的相似度可能是完全无关的
4. **缺少"我不知道"能力**: 不会主动说"没有相关信息"

### 1.5 向量检索的改进空间

**当前行为** (问题):
```
未知问题 → 强行匹配 → medium置信度 → 可能误导用户
```

**理想行为**:
```
未知问题 → 识别低相关性 → low置信度 → 明确告知"无相关信息"
```

**建议改进**:
1. 提高相似度阈值 (0.45 → 0.55)
2. 引入"无答案检测"机制
3. 使用多维度评分 (不仅仅是余弦相似度)

## 二、大语言模型层面的处理

### 2.1 工作原理

LLM接收以下输入:
1. **用户问题**
2. **检索到的上下文** (知识库内容)
3. **置信度标记** (high/medium/low)

然后生成回答。

### 2.2 Prompt设计分析

#### System Prompt (核心指令)

```
你是一个专业的加油机和充电桩设备故障诊断助手。

工作原则:
1. 根据提供的知识库内容回答用户问题
2. 如果知识库有明确答案，直接给出解决步骤
3. 如果知识库信息不完全匹配，可以基于相关信息推理，
   但要说明"以下建议基于相似故障经验，请谨慎操作"
4. 如果完全没有相关信息，诚实告知"暂无该故障的详细资料"，
   并提供通用排查建议
5. 涉及高压电、燃油等危险操作时，务必提醒安全注意事项
```

**关键点**:
- ✅ 要求诚实告知没有信息的情况
- ✅ 允许基于相关信息推理，但需要警告
- ✅ 要求安全提示

#### User Prompt (动态构建)

根据置信度不同，给LLM不同的提示:

| 置信度 | LLM提示 |
|-------|--------|
| `high` | "知识库中有该问题的明确解答，请直接回答" |
| `medium` | "知识库中有部分相关信息，请基于这些信息回答，并提醒用户如未解决可联系售后" |
| `low` | "知识库中暂无该问题的详细资料，请提供通用排查建议，并建议用户联系售后获取专业支持" |

### 2.4 LLM的优势与局限

#### ✅ 优势

1. **理解能力**: 能判断问题类型(故障 vs 概念)
2. **常识推理**: 基于常识回答概念性问题
3. **诚实性**: 能明确说"不知道"或"未明确提及"
4. **安全意识**: 提醒联系售后，不盲目指导
5. **灵活性**: 即使没有精确信息，也能给出合理建议

#### ❌ 局限

1. **依赖向量检索**: 如果检索给的上下文不对，LLM可能误判
2. **过度推理风险**: 可能基于相关但不准确的信息进行推断
3. **一致性问题**: 不同次调用可能给出不同程度的"承认不知道"
4. **成本**: 每次查询都需要调用API

### 2.5 LLM Prompt的改进空间

**当前Prompt**:
```
如果知识库信息不完全匹配，可以基于相关信息推理，
但要说明"以下建议基于相似故障经验，请谨慎操作"
```

**问题**: 这个指令可能导致LLM过度推理

**改进建议**:
```
如果知识库中没有直接相关的信息(相似度 < 0.6):
1. 明确告知用户"知识库中暂无该问题的记录"
2. 不要基于不相关的信息进行推理
3. 提供通用的排查建议
4. 建议联系专业技术支持

如果有部分相关信息(相似度 0.6-0.75):
1. 说明"以下是相关故障的处理方法，可能有帮助"
2. 给出参考信息
3. 提醒如果不匹配请联系售后
```

### 3.2 风险分析

#### 🔴 高风险场景

**向量检索强行匹配 + LLM过度推理**
```
用户: "设备冒烟了"
向量: 匹配到 "过热保护" (相似度0.55)
LLM: "可能是过热，建议..."
风险: 误导用户，可能造成安全问题
```

**解决方案**:
- 提高相似度阈值
- LLM Prompt中强调不确定时必须警告

#### 🟡 中风险场景

**相关但不准确的匹配**
```
用户: "屏幕颜色异常"
向量: 匹配到 "显示板检测错误"
LLM: 基于显示板问题推理
风险: 可能不是显示板问题，导致误判
```

**解决方案**:
- 要求LLM明确说明是推理，不是确定答案
- 加入免责声明

#### 🟢 低风险场景

**完全不相关的问题**
```
用户: "WiFi密码忘了"
向量: 虽然匹配到充电桩，但相似度中等
LLM: 识别出不是故障问题，诚实告知
风险: 低，LLM能正确处理
```

### 4.1 向量检索层改进

#### 方案1: 动态阈值调整

```python

# 改进
def get_dynamic_threshold(query_type):
    if has_error_code(query):
        return 0.70  # 错误代码查询要求更高
    else:
        return 0.55  # 一般查询适度提高
```

#### 方案2: 多维度评分

```python
def calculate_confidence(results):
    top_score = results[0]['similarity']
    score_gap = results[0]['similarity'] - results[1]['similarity']

    # 考虑分数绝对值和相对差距
    if top_score > 0.80 and score_gap > 0.15:
        return "high"  # 明显最优
    elif top_score > 0.60 and score_gap > 0.10:
        return "medium"
    else:
        return "low"  # 模糊不清
```

#### 方案3: 无答案检测

```python
def detect_no_answer(query, results):
    # 检测是否所有结果都低于可信阈值
    if all(r['similarity'] < 0.55 for r in results):
        return True, "知识库中可能没有相关信息"

    # 检测是否问题类型不匹配
    if is_concept_question(query) and all(is_fault_solution(r) for r in results):
        return True, "这似乎不是设备故障问题"

    return False, None
```

2. **诚实性**: 大多数情况下能说"不知道"
3. **通用建议**: 提供合理的通用排查步骤
4. **安全意识**: 建议联系售后，不盲目指导

#### ⚠️ 需要改进的地方

1. **向量检索阈值**: 0.45太低，导致误匹配
2. **相似度可靠性**: medium置信度范围太宽(0.45-0.75)
3. **LLM过度推理**: 有时基于不太相关的信息进行推断
4. **一致性**: 不同问题的处理标准不太一致

### 5.3 最佳实践建议

#### 对于系统开发者

1. **提高阈值**: 相似度 < 0.55 应该标记为low
2. **增强Prompt**: 更明确地指示LLM何时不要推理
3. **添加检测**: 实现"无答案检测"机制
4. **监控日志**: 记录low置信度的查询，持续优化

#### 对于用户

1. **提供完整信息**: 包括错误代码、设备型号等
2. **描述具体症状**: 越详细越容易匹配
3. **理性对待建议**: low/medium置信度的回答需谨慎
4. **及时反馈**: 帮助系统学习和改进
